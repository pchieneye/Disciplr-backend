# ETL Pipeline – Idempotent Batch Processing

## Overview

The ETL pipeline syncs on-chain Stellar transactions into the `transactions` table.
It runs on a configurable schedule via `ETLWorker` and delegates the actual work to
`TransactionETLService`.

---

## Idempotency guarantees

### 1. Batch-level deduplication (`etl_batches` table)

Every scheduled tick generates a **UUID batch ID** before any work starts.
`TransactionETLService.runETL(signal, batchId)` checks the `etl_batches` table:

| Scenario | Behaviour |
|---|---|
| `batch_id` not found | Create row (`pending`), proceed normally |
| `batch_id` found with `status = completed` | Return cached result immediately – **no re-processing** |
| `batch_id` found with `status = failed/running` | Row already exists; skip `create`, call `markRunning`, retry the work |

This means a crashed or retried run with the **same batch ID** will never
double-count analytics.

### 2. Row-level deduplication (`tx_hash` UNIQUE constraint)

`saveTransactions` uses `INSERT … ON CONFLICT (tx_hash) DO NOTHING`.
Even if the batch-level guard is bypassed (e.g. a manual backfill), individual
transactions are never inserted twice.

### 3. Input validation

`TransactionETLService.isValidTxHash(hash)` rejects any `tx_hash` that does not
match `/^[0-9a-fA-F]{1,128}$/`.  Rows with invalid hashes are skipped and counted
as `transactionsSkipped`, preventing SQL injection via ETL inputs.

---

## Batch lifecycle

```
pending → running → completed
                 ↘ failed
```

State transitions are conditional (`WHERE status NOT IN ('completed', 'failed')`)
so they are safe to call multiple times.

## `etl_batches` table schema

| Column | Type | Description |
|---|---|---|
| `batch_id` | UUID PK | Stable identifier generated by ETLWorker |
| `status` | enum | `pending / running / completed / failed` |
| `operations_fetched` | integer | Total Horizon operations fetched |
| `transactions_inserted` | integer | New rows written to `transactions` |
| `transactions_skipped` | integer | Rows skipped (duplicate `tx_hash` or invalid input) |
| `started_at` | timestamptz | When the batch transitioned to `running` |
| `finished_at` | timestamptz | When the batch reached a terminal state |
| `duration_ms` | integer | Wall-clock duration in milliseconds |
| `error_message` | text | Populated on failure |
| `created_at` | timestamptz | Row creation time |

## ETLWorker

`ETLWorker` manages the schedule and graceful shutdown:

- Calls `start(intervalMinutes?)` to begin periodic runs (default: 5 min).
- Generates a **fresh UUID** for each tick and passes it to `runETL`.
- On `stop()`: clears the interval, sends an `AbortSignal` to the in-flight run,
  and waits up to `drainTimeoutMs` (default 30 s) for it to finish.

```ts
import { ETLWorker } from './services/etlWorker.js'

const worker = new ETLWorker(config)
worker.start(5)          // run every 5 minutes
await worker.stop()      // graceful shutdown
```

---

## Retry behaviour

`ETLWorker` does **not** automatically retry a failed batch with the same ID.
Each new tick generates a new UUID, so a failed batch is simply superseded by
the next scheduled run.

For manual retries (e.g. ops tooling), pass the **same** `batchId` to
`runETL` – the service will skip the work if it already completed, or resume
from scratch if it failed.

---

## Security

- `tx_hash` values are validated against a hex-only regex before any DB write.
- No user-supplied strings are interpolated into raw SQL; all queries use
  Knex parameterised bindings.
- Export paths are not derived from ETL inputs, so path traversal is not
  possible.
